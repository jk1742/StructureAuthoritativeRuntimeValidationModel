6.5 Part 2 — Runtime Reconstruction in Component Environment

앞선 6.2 ~ 6.4의 Case 1, 2, 3은 DOM property 직접 변조와 unauthorized node 삽입 같은 runtime property and node forgery를 다루었다. 이 케이스들은 모두 *이미 존재하는 DOM*의 정합성 검증 문제에 해당한다.

Part 2에서는 검증 주제를 바꾼다. 본 절에서는 component 환경에서 subtree 자체가 runtime 도중 재조립되는 상황에서 정합성이 어떻게 유지되는지를 다룬다. 이는 4.2절에서 정의한 Module Lifecycle 및 Reconstruction Authority 개념의 실험적 검증에 해당한다. 현대 web application은 SPA(Single Page Application), component-based framework, server-driven UI update 같은 흐름을 통해 partial UI subtree를 반복적으로 mount, replace, update한다. 이러한 환경에서 reconstruction이 어떤 정합성 기준 위에서 수행되는가는 본 모델의 핵심 contribution 중 하나이다.

6.6 Case 4 : Component Reconstruction Consistency

본 케이스는 server-driven subtree reconstruction이 client에서 수행될 때, 기존 runtime state와 entity identity가 어떻게 보존되는지를 측정한다. 비교 대상은 현재 일반적으로 사용되는 reconstruction 방식 두 가지(generic baseline)와 본 모델의 identity reconciliation이다. 특정 framework 구현(React, Vue 등)이 아닌 architectural-level의 generic baseline으로 표기한다.

6.6.1 Compared Models

Model B와 Model C는 표면적으로 모두 식별자 기반 reconcile을 수행한다. 그러나 두 모델의 식별자는 본질적으로 다른 성격을 가진다. Model B의 key는 developer가 제공하는 매칭 hint이며, 동일 key를 두 개의 서로 다른 entity에 부여하는 것이 허용된다 (예: 의도치 않은 재사용). Model C의 entity id는 4.2절의 Reconstruction Authority가 발급하는 단방향 lineage 식별자이며, client가 임의로 발급할 수 없고 authority의 승인 record를 통해서만 형성된다. 본 케이스의 핵심 차이는 *식별자가 어디서 발급되는가*에 있다.

6.6.2 Scenarios

본 케이스는 component reconstruction의 두 가지 상호 보완적 요건을 모두 측정한다.

R1과 R2는 단순히 두 개의 독립 시나리오가 아니라 서로 반대 방향의 요건이다. R1은 동일 entity의 reconstruction에서 *재사용*을 요구하고, R2는 다른 entity의 reconstruction에서 *재사용 방지*를 요구한다. Reconstruction 모델이 두 요건을 모두 만족하려면, 단순한 형태 비교나 hint 기반 매칭으로는 부족하며, *식별자가 정확히 누가 동일하고 누가 다른가에 대한 권위 있는 기준*이 되어야 한다.

6.6.3 Detection Result

Model A (innerHTML swap)는 R1에서 사용자 입력을 손실한다. 모든 reconstruction이 완전 재생성이므로 entity identity 개념 자체가 존재하지 않는다. R2에서는 자연스럽게 입력값이 폐기되지만, 이는 *identity를 보존할 능력이 없기 때문에 우연히 정확한 결과*를 낸 것이다. 즉 Model A는 R2의 *outcome*은 만족시키지만 그 *이유*는 정합성과 무관하다.

Model B (keyed reconcile)는 R1에서 동일 key 매칭으로 입력값을 보존한다. 그러나 R2에서 developer가 동일 key를 유지하면 새 entity lineage가 시작되었음에도 잘못된 재사용(false reuse)이 발생한다. Model B의 key는 developer가 제공하는 hint이므로, lineage의 단절을 강제하지 못한다. 본 시나리오는 *"developer가 key를 다르게 부여하면 해결된다"*는 반론을 의식적으로 회피하지 않는다 — Model B의 한계는 *"key가 lineage를 표현하지 못한다"*는 architectural 성질이며, key 부여 방식에 의존한다는 것 자체가 본 논문이 지적하는 trust boundary 문제이다.

Model C (identity reconciliation)는 R1에서 entity id가 동일하므로 기존 node를 재사용하고 binding을 유지한다. R2에서는 authority가 새 entity id를 발급했으므로 기존 entity와의 매칭이 발생하지 않고 새 node가 생성된다. Entity id가 authority lineage의 일부이므로, R1과 R2의 구분은 client의 hint나 형태 비교가 아닌 *authority record*로부터 도출된다.

6.6.4 Reconstruction Time

Model A (innerHTML)는 native HTML parser에 reconstruction을 위임하므로 가장 빠른 결과를 보인다 (2,000 노드 기준 98.7 ms). 그러나 이는 *정합성 기준을 포기한 대가로 얻은 속도*이며, R1에서 사용자 입력을 모두 손실한다.

Model B와 Model C는 모두 O(N) reconcile 비용을 가지며, 측정 결과 2,000 노드 기준 약 11% 차이를 보인다. Model C는 WeakNodeMap rebind와 indexMap 조회를 추가로 수행하지만, 비용 차이는 일관되지 않은 작은 범위에 머무른다. 100 노드에서 일시적으로 큰 차이가 보이는 것은 측정 분산에 의한 것으로 판단되며, 실제 추세는 두 모델이 같은 차수에 머무름을 보여준다.

본 측정의 의의는 *"identity reconciliation이 가장 빠르다"*가 아니다. 핵심은 다음과 같다. Model B와 Model C는 reconstruction 비용이 사실상 동등하며, identity reconciliation은 Model B가 가지지 못하는 R2 보장을 *추가 비용 거의 없이* 제공한다.

6.6.5 Payload Size

HTML fragment 방식은 attribute syntax가 간결하여 JSON 형식보다 약 절반의 payload 크기를 가진다. Keyed와 Identity JSON은 거의 동일한 payload 크기를 보이며, identity id가 key보다 약간 짧은 식별자 형식이라 0.4 ~ 1% 차이가 발생한다. Payload 측면에서 Model A의 우위는 명확하다. 그러나 이 또한 정합성 기준의 부재와 trade-off 관계에 있다.

6.6.6 Discussion

본 케이스의 결과는 세 가지 관점에서 정리된다.

  • Combined consistency : 세 모델 중 identity reconciliation만이 R1과 R2를 동시에 만족한다. innerHTML swap은 R1을 만족할 수 없고, keyed reconcile은 R2를 만족할 수 없다. 이는 R1과 R2가 *서로 반대 방향의 요건*이라는 본질에 기인하며, 두 요건을 모두 만족하려면 *식별자가 권위 있는 lineage로 발급되어야 한다는 조건*이 필요함을 보여준다.
  • Performance trade-off : innerHTML은 가장 빠르나 정합성 기준이 없다. Keyed reconcile과 identity reconciliation은 유사한 reconstruction 시간을 가지며, identity의 R2 보장은 사실상 추가 비용 없이 얻어진다.
  • Payload : HTML fragment 방식이 JSON 두 형식보다 약 절반 크기로 가장 작다. 그러나 이는 정합성 기준의 완전 부재와 R1 손실의 대가이다. JSON 두 형식은 거의 동일한 크기이며, identity reconciliation은 keyed reconcile 대비 payload 측면에서 추가 비용이 없다.

6.6.7 Validity and Threats to Validity

  • Baseline의 단순성: 본 실험의 Model B (keyed reconcile)는 가장 단순한 형태의 reconcile 알고리즘이다. 실제 framework는 더 정교한 reconcile(예: longest common subsequence, fiber 기반 incremental update 등)을 사용한다. 그러나 R2의 한계는 reconcile 알고리즘의 정교함에서 비롯되는 것이 아니라 *식별자가 developer-provided hint라는 본질적 성질*에서 비롯되므로, 더 정교한 알고리즘이라도 동일 key를 부여한 경우의 R2 결과는 변하지 않는다.
  • R2 시나리오의 가정: R2는 developer가 key를 동일하게 유지한다고 가정한다. Developer가 의도적으로 다른 key를 부여하면 Model B도 정상 동작한다. 본 시나리오는 그러한 fallback을 부정하는 것이 아니라, *key 부여가 developer의 정확성에 의존한다는 trust boundary 문제*를 드러내는 것이다. Authority-issued entity id 는 이 의존성을 architectural 단계에서 제거한다.
  • Reconstruction 비용의 환경 의존성: JSDOM 환경의 측정 결과이며, 실제 브라우저의 layout/paint 비용은 포함되지 않는다. 그러나 R1/R2의 outcome은 reconstruction 알고리즘 자체의 성질에 의존하므로 환경에 무관하다.
  • 단일 시나리오 형태: 본 실험은 form-like flat structure 위에서 측정되었다. Deep nested component, sibling reordering, conditional rendering 같은 더 복잡한 시나리오는 다루지 않는다. 향후 연구로 nested reconstruction의 정합성 분석을 확장할 수 있다.

6.7 Chapter Conclusion

본 장은 두 부분으로 구성되었다. Part 1 (6.2 ~ 6.4)에서는 runtime property and node forgery를 다루었으며, DOM 외부에 위치한 validation reference layer가 진실값 위조 evasion에 대해 분리 효과를 가짐을 Case 1과 Case 3에서 확인하였고, Case 2는 코멘트로 처리하였다. Part 2 (6.5 ~ 6.6)에서는 component 환경의 runtime reconstruction 정합성을 다루었으며, identity reconciliation이 기존 일반적 접근(innerHTML swap, keyed reconcile) 모두가 동시에 만족하지 못하는 R1 + R2 요건을 만족함을 확인하였다.

Part 1과 Part 2는 서로 다른 검증 주제를 다루지만, 본 모델의 일관된 architectural principle인 *외부 권위 있는 식별자 lineage를 통한 정합성 검증*이라는 점에서 공통된 기반을 가진다. Part 1에서는 이 lineage가 진실값 검증의 reference 역할을 하고, Part 2에서는 이 lineage가 reconstruction의 정합성 기준 역할을 한다.
