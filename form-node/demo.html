Structure-Authoritative Runtime Validation — form demo

On load, the canonical entity tree is fetched and registered outside the DOM. Two properties are shown: (1) the structure is authoritative — node type, parent, order, and child count must hold, while text, class, and style may change freely; (2) the input's value becomes truth only through the approved interaction path (commit), so a direct DOM forgery is detected. Tamper as an attacker, then press Validate.

// C) Free to change: text / class / style             -> VALID
document.querySelector('label').textContent = 'Login ID';
document.querySelector('#username').style.color = 'red';

// D) Structure tampering: insert / remove / replace   -> STRUCTURAL_DEVIATION
document.querySelector('#login').appendChild(document.createElement('input'));

What "canonical entity" is here — and what this demo does not claim

The canonical entity tree is loaded from canonical.json, which stands in for an authority-issued entity-id lineage. In a real deployment that lineage originates from an external Reconstruction Authority; in this demo it is a static stand-in, not a live authority.

Each entity.id is an authority-issued id carried in canonical.json — it is not the DOM id attribute. Validation trusts the binding between a node and its canonical entity, not the node's tag or shape.

Scope. This demo shows detection after the registry is established: you tamper from the console after load, then press Validate. It does not attempt to prove that the authority is external or that its delivery is tamper-proof — those are stated trust assumptions in the paper (the authority is a precondition, not what the model protects). Load-time or delivery tampering (for example, intercepting the fetch) is out of scope.
