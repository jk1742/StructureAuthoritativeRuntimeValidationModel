6. Experimental Evaluation

6.1 Experimental Environment

실험은 동일한 browser runtime 환경에서 수행하였다. Node.js v22 환경에서 JSDOM v25 기반의 browser-like runtime 위에서 측정을 수행하였으며, validation time과 reconstruction time 측정은 performance.now() high-resolution timer를 사용하였다. 각 측정 항목은 warmup 2회, 본 측정 7회 평균을 사용하였다.

본 장의 실험에서 비교 대상으로 사용된 baseline 모델들은 현재 실제로 사용되는 검증 및 reconstruction 기법 (DOM-internal validation metadata, MutationObserver, snapshot diff, innerHTML swap, keyed reconcile)의 가장 단순한 형태를 직접 구현한 것이다. 제안 모델(Proposed)은 본 논문 3장 및 4장에서 정의한 Structure-Authoritative Runtime Architecture의 reference implementation이며, baseline과 동일한 fairness 조건을 위해 framework-specific 최적화 계층을 포함하지 않은 최소 구조로 구현하였다. 따라서 본 실험은 특정 framework 간 성능 비교가 아니라 architecture-level 메커니즘 비교에 해당한다.

본 장은 두 부분으로 구성된다. Part 1 (6.2 ~ 6.4)은 DOM property 및 node 변조 탐지에 관한 케이스를 다루며, validation reference layer의 위치가 trust boundary에 미치는 영향을 검증한다. Part 2 (6.5 ~ 6.6)은 component 환경의 runtime reconstruction 정합성에 관한 케이스를 다루며, component subtree의 재조립에서 entity identity가 어떻게 보존되는지를 검증한다. Part 1과 Part 2는 서로 다른 검증 주제를 다루지만, 본 모델의 일관된 architectural principle인 *외부 권위 있는 식별자 lineage를 통한 정합성 검증*이라는 점에서 공통된 기반을 가진다.

Part 1 — Runtime Property and Node Forgery

6.2 Case 1 : Runtime State Forgery Detection

본 케이스는 3.3절 Runtime State Forgery Boundary에서 제안한 검증 모델이, DOM 내부에 validation metadata를 삽입하는 기존 접근과 비교하여 외부 mutation에 대한 분리 효과를 가지는지를 검증한다.

6.2.1 Compared Models

실험은 두 가지 비교 대상 모델과 한 가지 제안 모델로 구성된다.

여기서 한 가지 짚어야 할 점이 있다. Baseline 계열 모델은 <meta> tag를 노드 수만큼 추가로 생성해야 한다. 이는 baseline의 단순한 구현 한계가 아니라 DOM 내부에 진실값을 두는 모든 접근이 공유하는 구조적 비용이다. Hidden input, data-* attribute, comment node 등 어떤 저장 방식을 선택해도 본질적인 비용 구조는 동일하다. 따라서 본 비교는 baseline에 본래 존재하지 않는 부담을 임의로 부과한 것이 아니라, 해당 접근 방식이 본질적으로 가지는 비용을 그대로 측정한 결과이다.

6.2.2 Attack Scenarios

공격자는 browser console 또는 injected script를 통해 DOM property를 직접 변경할 수 있다고 가정한다. 본 케이스에서는 다음 네 가지 forgery 시나리오를 정의한다.

S1~S3은 property를 직접 변경하는 가장 단순한 공격이며, 정상 interaction lifecycle을 거치지 않는다. S4는 본 모델이 식별 대상으로 삼는 핵심 위협이다. 검증 기준이 DOM 위에 존재한다면 공격자는 진실값 자체도 위조할 수 있으며, 이 경우 dom value === truth value가 성립해 단순 equality 검사는 통과한다.

6.2.3 Detection Result

여섯 개의 시나리오 인스턴스 (S1, S2, S3, S4-a/b/c) 각각에 대해 두 모델의 탐지 결과를 측정하였다.

S1~S3과 같은 단순 forgery는 두 모델 모두에서 탐지되었다. 진실값과 현재값이 서로 다르기 때문에 검증 기준의 위치와 무관하게 equality 검사만으로도 차이가 드러난다. 그러나 S4 evasion에서는 두 모델의 결과가 명확하게 갈렸다. Baseline은 meta tag 자체가 DOM의 mutable execution surface 위에 위치하므로, 공격자가 setAttribute("content", ...)를 통해 meta tag를 함께 위조할 수 있다. 그 결과 node.value와 meta.content가 동일해져 검증을 통과해 버린다 (Bypassed). 반면 Proposed 모델의 indexMap과 WeakNodeMap은 application 내부 closure scope에 위치하므로 외부 script가 직접 손댈 수 없는 영역이며, DOM mutation API만으로는 진실값이 오염되지 않는다.

6.2.4 Runtime Overhead - Validation Time

3.3절은 Proposed 모델이 외부 registry lookup을 추가로 수행함을 명시한다. 본 절에서는 노드 수가 증가함에 따라 두 baseline 모델과 Proposed 모델의 검증 비용이 어떻게 변화하는지 측정한다. 실험은 10, 50, 100, 200, 500, 1000, 2000 개의 input node를 등록한 동일한 form 구조 위에서 수행되었다.

Baseline-naive는 querySelector가 검증 호출마다 전체 문서를 스캔하므로 노드 수에 따라 O(N²) 패턴으로 폭증한다. 1,000 노드에서 약 2.87초가 소요되어 본 실험의 범위에서는 2,000 노드 측정을 생략하였다. 단, 이는 baseline 모델 자체의 본질적 한계가 아니라 단순한 구현 선택의 결과임을 명시한다.

보다 공정한 비교는 사전 인덱싱을 사용하는 Baseline-optimized와 Proposed 사이에서 이루어진다. 두 모델은 2,000 노드 기준 각각 7.6 ms와 8.6 ms로 측정되었으며 약 12%의 추가 비용이 관찰되었다. 이는 Proposed 모델이 WeakNodeMap.get 단계를 한 번 더 거치기 때문이며, 모든 측정 구간에서 두 모델은 동일한 O(N) 차수를 유지한다. 결과적으로 검증 시간 관점에서 Proposed의 추가 비용은 작고 예측 가능한 수준이다.

6.2.5 Runtime Overhead - Document Payload Size

Proposed 모델이 DOM 내부에 metadata를 삽입하지 않는다는 점은 직렬화된 문서의 크기에도 직접적으로 반영된다. 동일한 노드 수 조건에서 baseline은 <meta> tag를 추가로 포함해야 하므로 노드 수에 비례한 추가 payload가 발생한다.

Baseline은 노드 수가 증가할수록 payload overhead가 수렴적으로 약 94% 수준에 도달한다. 이는 노드당 약 47~48 bytes의 <meta> 선언이 누적된 결과이며, 네트워크 전송 비용과 초기 parse cost 양쪽에 비례한 부담이 된다. Proposed 모델은 진실값을 application 메모리에 보관하므로 직렬화 문서에는 추가 비용이 발생하지 않는다.

6.2.6 Discussion

본 케이스의 결과는 세 가지 관점에서 정리할 수 있다.

  • Forgery detection : Baseline 50.0%, Proposed 100.0%. 단순 forgery는 두 모델 모두 탐지 가능하지만, evasion forgery에서는 검증 기준이 DOM 위에 있는 baseline이 우회된다. 이는 3.3절이 명시한 "검증 기준이 DOM과 동일한 mutation surface에 놓이는 경우의 한계"가 실제로 발현된 결과이다.
  • Validation time : 사전 인덱싱을 사용하는 baseline 대비 Proposed는 약 12% 추가 비용을 보인다. 두 모델 모두 O(N) 차수를 유지하며, registry lookup으로 인한 비용은 작고 예측 가능한 수준이다.
  • Payload size : Proposed는 baseline 대비 약 94% 작은 문서 크기를 유지한다. 이는 DOM 내부에 검증 metadata를 삽입하지 않는다는 본 모델의 구조적 선택이 직렬화 단계에서도 그대로 반영됨을 보여준다.

종합하면, Proposed 모델은 baseline 대비 payload는 줄이고(saving), 검증 비용은 거의 동등(parity)하며, 보안성은 향상(gain)되는 trade-off를 가진다.

6.2.7 Validity and Threats to Validity

본 케이스의 실험 결과를 해석할 때 다음과 같은 한계를 함께 고려해야 한다.

  • Baseline 저장 매체의 일반성: 본 실험은 진실값 저장 매체로 <meta> tag를 사용하였다. Hidden input, data-* attribute, comment node 등 다른 매체를 사용하더라도 모두 DOM mutable surface 위에 위치하므로 S4 evasion에 대한 결과는 동일할 것으로 예상되지만, 본 실험은 직접 측정하지 않았다.
  • 공격자 가정의 단계성: S4 evasion은 공격자가 검증 구조를 알고 있다고 가정한다. DOM 내부 metadata는 외부 script에서 전부 읽을 수 있으므로 이 가정은 Kerckhoffs 원칙 관점에서 합리적이다. 그러나 closure scope의 registry 역시 prototype pollution이나 module loader 단계의 공격에는 노출될 수 있으며, 이는 본 모델의 trust assumption 안에서는 application runtime을 trusted로 가정하는 것에 의존한다.
  • 측정 환경의 외부 타당성: 본 실험은 JSDOM 환경에서 수행되었다. 실제 브라우저 환경의 layout, reflow, GC 영향과는 차이가 있을 수 있다. 다만 두 모델 모두 동일한 환경에서 측정되었으므로 상대 비교의 의미는 유지된다.
  • 단일 케이스의 일반화 한계: 본 케이스는 단일 검증 기준이 DOM 외부에 위치할 때의 효과를 보였을 뿐, 본 모델이 모든 공격에 대해 강건함을 보인 것은 아니다. 후속 케이스에서 MutationObserver 기반 접근, Virtual DOM 유사구현, Proposed 모델 단독 평가, 그리고 진실값 state의 server-client serialization 환경 에 대한 추가 실험을 수행한다.
