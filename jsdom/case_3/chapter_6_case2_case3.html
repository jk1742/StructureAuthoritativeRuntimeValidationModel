6.3 Case 2 : MutationObserver Baseline (Comment)

MutationObserver(이하 MO)는 DOM tree와 attribute 변경을 변경 시점에 표적 감시할 수 있는 browser 내장 API이다. 변경된 객체만 callback으로 전달되므로 전체 트리를 매번 비교하는 방식 대비 성능 측면에서 분명한 우위를 가진다. 본 케이스에서는 이러한 성능 우위를 인정하되, MO가 본 모델이 제안하는 분리 효과를 제공할 수 없는 이유를 정리한다. 성능 비교 실험은 수행하지 않으며, 회피 시나리오 한 가지로 한계를 보인다.

6.3.1 MutationObserver Bypass Scenario

MO는 application 내부 API가 아니라 browser가 외부 script에도 동일하게 노출하는 표준 API이다. 공격자는 동일한 권한으로 다음과 같이 감시 자체를 중단할 수 있다. 


  // application setup
  const obs = new MutationObserver((records) => {
      // detect property mutation
  });
  obs.observe(input, { attributes: true, attributeFilter: ["value"] });

  // attacker script (same browser runtime)
  obs.disconnect();              // 감시 중단
  input.value = "attacker";      // 변조 (기록 누락)
  obs.observe(input, { attributes: true, attributeFilter: ["value"] });
                                 // 재감시 등록 시점부터 다시 정상 동작

공격자가 application 내부의 obs 참조를 획득하지 못하더라도, MutationObserver.prototype.disconnect를 통한 일괄 회피가 가능하며, callback function 자체를 Function.prototype 또는 MutationObserver 생성자 단계에서 가로채는 방식도 가능하다. 보다 본질적인 한계는 이러한 구체적 회피 기법의 존재 여부가 아니라, MO가 신뢰 기반을 DOM mutation surface 자체에 두고 있다는 점이다.

6.3.2 Conclusion on Case 2

MO는 변경 시점에 표적 감시가 가능하므로 성능 측면의 우위는 분명하다. 그러나 MO 자체가 DOM과 동일한 mutation surface 위에서 동작하는 API이며, 검증 기준 역시 DOM 위에 머무른다. 따라서 6.2의 Baseline (meta tag) 모델과 동일한 trust boundary 문제를 공유하며, 본 모델이 제안하는 DOM-외부 validation reference layer의 분리 효과를 대체할 수 없다. 이러한 이유로 본 케이스는 정량 성능 비교 대상에서 제외하였다.

6.4 Case 3 : Snapshot Diff vs Identity Continuity

본 케이스는 본 모델의 identity continuity 개념(3.2절 stale subtree 정의 참고)이 기존 snapshot 기반 접근으로는 식별하기 어려운 상황을 탐지할 수 있음을 보인다. 비교 대상은 단순 snapshot diff 방식으로, Virtual DOM reconcile의 가장 기본적인 형태이다. 본 실험에서는 구조적으로 최소한의 두 모델만 사용하며, framework-specific 최적화 계층은 사용하지 않는다.

6.4.3 Detection Result

T1은 본 케이스의 핵심 시나리오이다. 공격자가 기존 node를 제거하고 완전히 동일한 tag, attribute, value를 가진 신규 node로 교체한 경우, snapshot의 평면 표현은 변경 전후가 동일하므로 diff에서 어떠한 mismatch도 발견되지 않는다. 그러나 새로 삽입된 node는 등록 시점의 entity와 binding되어 있지 않으므로 WeakNodeMap에서 entity id를 얻을 수 없고, 본 모델은 이를 identity 단절로 판정한다. 이는 3.2절에서 정의한 stale subtree 및 unauthorized subtree replacement 상황에 해당한다.

T2 대조군에서는 두 모델 모두 valid로 정상 통과하였다. T3과 T4의 단순 mutation 시나리오에서는 두 모델 모두 탐지에 성공하였다. 이를 통해 identity continuity 접근이 기존 snapshot-diff가 제공하는 탐지 능력을 유지하면서, snapshot-diff가 놓치는 상황을 추가로 탐지함을 확인할 수 있다.

6.4.4 Performance Comparison

Identity 접근이 추가 탐지 능력을 제공하는 대신 성능 비용이 어떻게 변하는지 보조 측정으로 함께 비교하였다.

두 모델은 모든 측정 구간에서 거의 동일한 검증 비용을 보였다 (2000 노드 기준 약 1.01배 차이). Snapshot-diff는 매 검증마다 전체 트리를 재직렬화하고 path 단위 비교를 수행하며, identity 접근은 동일한 깊이의 walk와 노드별 WeakNodeMap lookup을 수행한다. 두 방식 모두 노드 수에 비례한 O(N) walk 비용이 지배적이므로 성능 차이는 미미하다.

본 측정의 핵심은 성능 우열의 입증이 아니라, identity continuity 접근이 snapshot-diff 대비 유의미한 성능 손실 없이 추가 탐지 능력을 제공함을 확인하는 데 있다. 특히 측정에서 보이는 2000 노드 기준 154 µs/node 수준의 절대 비용은 JSDOM 환경의 walk 비용을 포함한 결과이며, 실제 브라우저 환경에서는 더 낮은 절대 비용이 기대된다.

6.4.5 Discussion

  • Detection capability : T1 identical-form replacement에서 snapshot-diff는 false negative를 발생시키지만 identity continuity는 탐지한다. 이는 본 모델의 핵심 정의(3.2절 stale subtree, 4.1절 identity continuity)가 실제로 식별 가능한 상황을 만들어냄을 보여준다.
  • Performance parity : 두 모델은 동일 차수의 walk 비용을 가지며, 측정 범위 내에서 차이는 1% 내외이다. 즉 추가 탐지 능력은 사실상 추가 비용 없이 제공된다.
  • 모델 단순성 : 본 비교에서 식별된 차이는 framework 단의 추가 최적화에 의존하지 않으며, 최소 구조의 closure-scope registry만으로 발생한다.

6.4.6 Validity and Threats to Validity

  • Snapshot 표현의 단순화: 본 실험의 snapshot은 tag, value, path만 기록하는 최소 표현을 사용한다. 더 정교한 snapshot이 element 식별자나 random nonce를 함께 기록한다면 T1의 false negative는 줄어들 수 있다. 그러나 그러한 nonce 자체가 DOM 내부에 위치하는 한 6.2의 Baseline (meta tag) 모델과 동일한 trust boundary 문제를 가지게 된다.
  • 실제 reconcile 알고리즘과의 차이: 본 실험은 production-grade Virtual DOM의 key-based reconcile를 직접 비교하지 않는다. key-based reconcile은 node 이동을 추적할 수 있으나, 본 케이스의 T1과 같이 동일 key로 교체된 노드에 대해서는 동일하게 false negative가 발생한다.
  • 성능 측정 환경: Case 1과 동일하게 JSDOM 환경의 측정이며, 절대 수치보다 두 모델의 상대 비교가 의미를 가진다.
